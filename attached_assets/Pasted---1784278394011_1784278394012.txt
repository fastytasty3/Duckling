Доработай существующую программу. Не создавай новый проект, не удаляй и не ломай уже реализованные функции.

Главная доработка: создай защищенный вход для контролирующего лица с обязательным логином и паролем.

1. Отдельный вход для контролирующего лица

На экране входа должно быть два варианта:

1. «Войти как оператор».
2. «Войти как контролирующее лицо».

При выборе «Войти как контролирующее лицо» обязательно запрашивать:

* логин;
* пароль.

Без правильного логина и пароля доступ к панели контролирующего лица невозможен.

Нельзя открывать панель контролирующего лица через прямую ссылку, сочетание клавиш, изменение адреса страницы или кнопку «Назад».

После перезапуска программы контролирующее лицо должно авторизоваться повторно, если не включена функция «Запомнить вход».

2. Безопасное хранение паролей

Пароли нельзя хранить в базе данных в открытом виде.

Требования:

* хранить только хеш пароля;
* использовать bcrypt или argon2;
* использовать индивидуальную соль;
* не показывать пароль в журнале действий;
* не передавать пароль в незашифрованном виде между компонентами приложения;
* поле пароля должно скрывать введенные символы;
* добавить кнопку показа и скрытия пароля;
* ограничить количество неудачных попыток входа.

После 5 неправильных попыток:

* временно заблокировать вход на 5 минут;
* записать событие в журнал безопасности;
* показать сообщение: «Слишком много неудачных попыток. Повторите вход позже».

Не сообщать, что именно введено неправильно — логин или пароль. Показывать общее сообщение:

«Неверный логин или пароль».

3. Учетная запись контролирующего лица

Для пользователя с ролью контролирующего лица хранить:

* ФИО;
* логин;
* хеш пароля;
* роль;
* подразделение;
* дату создания;
* дату последнего входа;
* статус учетной записи;
* количество неудачных попыток входа;
* время временной блокировки;
* признак обязательной смены пароля;
* активен или заблокирован.

Добавь статусы:

* активен;
* временно заблокирован;
* заблокирован администратором;
* требуется смена пароля.

4. Первый вход

При первом запуске программы предложить создать главную учетную запись администратора или контролирующего лица.

Не использовать стандартные логины и пароли вроде:

* admin;
* administrator;
* 1234;
* 123456;
* password.

Пользователь должен самостоятельно задать:

* ФИО;
* логин;
* пароль;
* подтверждение пароля.

Требования к паролю:

* минимум 8 символов;
* минимум одна заглавная буква;
* минимум одна строчная буква;
* минимум одна цифра;
* желательно один специальный символ.

После создания первой учетной записи не показывать этот мастер повторно.

5. Панель контролирующего лица

После успешного входа открывать отдельную панель «Контроль рабочих столов».

Контролирующее лицо должно видеть все зарегистрированные рабочие столы в реальном времени.

Для каждого стола отображать:

* номер рабочего стола;
* наименование или расположение;
* авторизован ли оператор;
* ФИО оператора;
* табельный номер;
* смену;
* время входа;
* текущий статус;
* текущий штрихкод;
* артикул;
* наименование товара;
* количество;
* время начала операции;
* текущую длительность операции;
* время паузы;
* время последнего сканирования;
* количество обработанных единиц за смену;
* количество завершенных операций;
* среднее время обработки единицы.

Статусы рабочих столов:

* «Не авторизован»;
* «Готов к работе»;
* «Товар в работе»;
* «Пауза»;
* «Нет активности»;
* «Слишком долгая операция»;
* «Ошибка связи»;
* «Смена завершена».

Использовать цветовое выделение:

* зеленый — работа идет нормально;
* желтый — длительная операция или нет активности;
* красный — ошибка, превышение критического времени или зависшая операция;
* серый — стол не авторизован.

Информация должна обновляться автоматически без ручного обновления страницы.

6. Доступ контролирующего лица

Контролирующее лицо может:

* просматривать все рабочие столы;
* видеть текущую работу всех операторов;
* просматривать историю операций;
* просматривать статистику по столам;
* просматривать статистику по сотрудникам;
* просматривать статистику по товарам;
* фильтровать данные;
* выгружать данные в Excel;
* добавлять комментарии;
* отмечать проблемные операции;
* завершать зависшие операции;
* просматривать журнал действий;
* менять собственный пароль;
* выходить из учетной записи.

Контролирующее лицо не должно автоматически получать права администратора.

Без отдельного разрешения контролирующее лицо не может:

* удалять пользователей;
* менять роли пользователей;
* восстанавливать базу данных;
* удалять историю операций;
* менять технические настройки программы;
* просматривать пароли;
* сбрасывать чужие пароли.

7. Общая база данных

Важно: контролирующее лицо должно видеть данные со всех рабочих компьютеров и столов.

Локальная отдельная SQLite-база на каждом компьютере для этой задачи недостаточна.

Переведи программу на клиент-серверную архитектуру:

* рабочие столы являются клиентами;
* данные передаются на центральный сервер;
* контролирующая панель получает данные с центрального сервера;
* все компьютеры работают в одной локальной сети;
* программа должна работать внутри локальной сети предприятия без доступа в интернет.

Используй:

* серверную часть на Node.js и TypeScript;
* PostgreSQL как центральную базу данных;
* REST API или WebSocket;
* WebSocket для обновления текущего состояния столов в реальном времени;
* Electron-приложение на рабочих столах;
* отдельную защищенную панель контролирующего лица.

Если центральный сервер временно недоступен:

* рабочий стол продолжает работать;
* операции временно сохраняются локально;
* показывается статус «Нет связи с сервером»;
* после восстановления связи данные автоматически синхронизируются;
* операции не должны дублироваться;
* каждой операции присваивать уникальный UUID.

Формат файла — .xlsx.

9. Оформление Excel

Сделай Excel-отчет аккуратным и готовым для работы:

* закрепить верхнюю строку;
* включить автофильтры;
* автоматически подобрать ширину колонок;
* оформить заголовки;
* даты сохранять как даты Excel;
* время сохранять как время и длительность, а не как обычный текст;
* проценты сохранять числовыми значениями с процентным форматом;
* длительность более 24 часов отображать в формате [ч]:мм:сс;
* добавить итоговые строки;
* не объединять ячейки внутри рабочих таблиц;
* добавить условное форматирование превышения нормативов.

Название файла:

Контроль_рабочих_столов_ГГГГ-ММ-ДД_ЧЧ-ММ.xlsx

Для отчета за смену:

Отчет_смена_День_2026-07-17.xlsx

11. Завершение сессии

В панели контролирующего лица должна быть заметная кнопка «Выйти».

Автоматически завершать сессию после настраиваемого периода бездействия, например через 30 минут.

За 2 минуты до выхода показать предупреждение:

«Сессия скоро завершится из-за отсутствия активности».

После выхода возвращать пользователя на экран ввода логина и пароля.

При сворачивании программы не завершать сессию сразу.

12. Изменение пароля

Добавь раздел «Безопасность».

Контролирующее лицо может изменить свой пароль, указав:

* текущий пароль;
* новый пароль;
* подтверждение нового пароля.

Новый пароль не должен совпадать с текущим.

После смены пароля:

* завершить другие активные сессии этого пользователя;
* записать действие в журнал безопасности;
* потребовать повторный вход.

13. Восстановление доступа

Сброс пароля контролирующего лица может выполнять только администратор.

Администратор не должен видеть старый пароль.

При сбросе:

* назначается временный пароль;
* при следующем входе пользователь обязан установить новый пароль;
* событие записывается в журнал безопасности.

Если в системе только одна учетная запись администратора, создай защищенный механизм аварийного восстановления через локальный сервер, но не через открытый секретный код в интерфейсе.

14. Проверка прав на сервере

Все права доступа проверять не только в интерфейсе, но и на серверной части.

Запрещено просто скрывать кнопки.

Каждый серверный запрос должен проверять:

* авторизован ли пользователь;
* не истекла ли сессия;
* активна ли учетная запись;
* есть ли у пользователя нужная роль и разрешение.

При попытке доступа без прав возвращать ошибку «Доступ запрещен».
