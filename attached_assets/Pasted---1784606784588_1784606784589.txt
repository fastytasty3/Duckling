В коде секреты не хранятся
Использование принципов SOLID, KISS, DRY, YAGNI.
Функции и методы меньше 30-50 строк
Максимальная вложенность 3-4 уровня (вложенность функций, в плане условий, циклов)
Использовать линтеры и формартировщики при разработке
Должны быть реализованы автотесты для кода.
Использование моки и тестовых данных
Swagger/OpenAPI для API использовать, чтобы была документация проекта. Без этого никак.
README.md
 в каждом модуле/сервисе, должно быть четкое описание сервисов
Rate limiting настроены в проектах
Реализован RBAC, должно быть ограничение доступов и авторизация
Никакого http только https
Проводить сканы на безопасность (sast/sca). Если npm можно npm audit для Node.js делать. Инструменты зависят от того, что используете
В репозиториях должны быть настроены protected branches. Прямой push в main/master запрещён. Изменения должны проходить через merge request / pull request. Два человека минимум должны делать ревью
Запрещено логировать пароли, токены, refresh/access token, cookie, API key, коды восстановления, одноразовые коды, персональные данные без необходимости.
Ошибки приложения не должны раскрывать пользователю stack trace, внутренние пути, SQL-запросы, названия таблиц, переменные окружения, версии библиотек и внутреннюю структуру сервиса.
Все входные данные должны валидироваться на стороне сервера.
Для входных параметров API должны быть заданы типы, допустимые значения, ограничения длины, формата, диапазона и обязательности.
Для работы с БД должны использоваться параметризованные запросы, ORM или безопасные query builder. Конкатенация пользовательского ввода в SQL-запросы запрещена.
Для API необходимо реализовывать не только RBAC, но и проверку доступа к конкретному объекту. Пользователь не должен получать доступ к чужим объектам через подмену id, uuid, slug или другого идентификатора.
Для каждого endpoint должна быть понятна модель доступа: кто может вызвать метод, при каких условиях, к каким объектам и с какими действиями.
Неиспользуемые, тестовые, debug и deprecated endpoint должны быть удалены или закрыты.
Для cookie должны использоваться безопасные флаги: Secure, HttpOnly, SameSite. Сессионные cookie не должны быть доступны из JavaScript без необходимости
Для JWT и иных токенов должны быть заданы срок жизни, проверка подписи, проверка issuer/audience при необходимости, безопасное хранение и механизм отзыва/обновления.
Пароли пользователей нельзя хранить в открытом виде или шифровать обратимым способом. Для хранения паролей должны использоваться стойкие алгоритмы хэширования паролей с солью.
Для загрузки файлов должны быть ограничения по размеру, типу, расширению, содержимому и месту хранения. Загружаемые файлы нельзя исполнять на сервере.
Имена загружаемых файлов не должны напрямую использоваться как путь на сервере. Необходимо исключить path traversal и перезапись существующих файлов.
При работе с внешними URL, webhooks, callback и интеграциями необходимо учитывать риск SSRF. Запрещено без проверки обращаться к внутренним адресам, localhost, metadata-сервисам и служебным сетям.
CORS должен быть настроен строго. Запрещено использовать wildcard * для чувствительных API, особенно если используются cookie или авторизационные заголовки.
Для веб-приложений должны использоваться security headers: HSTS, Content-Security-Policy, X-Content-Type-Options, Referrer-Policy и другие применимые заголовки.
Все окружения должны быть разделены: dev, test, stage, prod. Тестовые данные и тестовые доступы не должны использоваться в production.
Никаких уз по умолчанию из разряда admin:admin. Выставить валидацию создания pass
Зависимости должны быть зафиксированы в lock-файлах. Необходимо регулярно обновлять библиотеки и устранять уязвимости из SCA-отчётов.
